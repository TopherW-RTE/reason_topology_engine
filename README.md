# Reasoning Topology Engine
[![DOI](https://zenodo.org/badge/DOI/10.5281/zenodo.20149576.svg)](https://doi.org/10.5281/zenodo.20149576)

Multi-model consensus reasoning system that scores, synthesizes, and
persistently accumulates reasoning traces into topology scaffolds — improving
without retraining. Single-sided engine extracted from a larger hybrid system.
Apache 2.0.

## Abstract

Current large language model evaluation is in crisis. Monolithic benchmarks
assess final answers, not reasoning quality. A wave of new work now targets
the reasoning trace itself — scoring it, curating it, retrieving it. But
nearly all of these efforts treat human input as an external prompt and AI
output as the sole artifact worth evaluating. This leaves a critical
structural gap: the human's own reasoning is never systematically extracted,
scored, and fused with the machine's reasoning into a single, persistent
topology.

This repository introduces the core component of a system designed to close
that gap: a Six-Gate Fractal Engine for non-parametric reasoning-trace
evaluation. The engine scores any reasoning trace — human or machine — across
six information-theoretic dimensions: Information Cost, Signal Entropy,
Variety Adequacy, Throughput Efficiency, Flow Resistance, and Collective
Coherence. It is model-agnostic, fully auditable, and operates without weight
updates. High-scoring traces are inserted into a self-improving reasoning
library; low-scoring traces are rejected, preventing memory pollution.

The engine is presented here in its single-sided form, applied to traces
generated by large language models. It already functions as a standalone
quality-control layer for any reasoning-trace pipeline.

But this architecture was designed from the ground up to be symmetric. The
same six-gate scoring applies, without modification, to the structured
extraction of a human's own logic trace before it reaches the language model.
When both sides are scored and combined, the result is not a chat log. It is
a bidirectional co-reasoning topology — a living, updatable lattice of human
and machine thought that deepens with every interaction. A system that does
not merely answer, but actively fosters and preserves the user's intent.

This release documents the single-sided engine, its architectural role, and
the calibration methodology underway. The bidirectional extension is under
active development and will follow in a subsequent release. What is made
public today is the foundation. The blueprint is visible.

> A multi-model consensus system that scores, synthesizes, and persistently
> accumulates reasoning traces into reusable topology scaffolds — getting
> measurably better at reasoning over time without retraining.

## What This Is

This is a standalone Python implementation of a novel reasoning architecture:
instead of asking one language model a question and accepting its answer, it
asks multiple independent models in parallel, scores the collective reasoning
through a consensus pipeline, synthesizes the result into a persistent
**Reasoning Knowledge Graph (RKG)**, and uses that graph as a structural
scaffold for every future response to the same class of question.

The system does not improve through fine-tuning or retraining. It improves
through accumulation — each run leaves a better-informed topology behind for
the next one.

This repository is a targeted extraction of the most conceptually novel
component of a larger research system. It runs fully locally with no external
API dependencies beyond Ollama.

## The Core Idea

Current LLM deployments treat each inference as a stateless event. The model
processes a prompt, returns a response, and retains nothing. Every run starts
from the same prior.

This system treats inference differently. Each response is a **reasoning trace**
— a record not just of what was said, but of the logical structure
used to say it. Those traces are scored for surprise and consensus, synthesized
into a topology, and stored. The next time a structurally similar question
arrives, the accumulated topology is injected as a hypothesis scaffold —
constraining the reasoning space toward what prior runs found reliable.

The result is a system that converges. Given enough runs on a prompt class,
the collective reasoning stabilizes into a high-confidence topology that
reflects genuine multi-model consensus rather than any single model's prior.

## How It Works

```
1. LEDGER SEARCH      — semantic similarity search for prior topology
                        matching the incoming prompt class

2. TRACE COLLECTION   — prompt sent to slot_a, slot_b, slot_c sequentially
                        each model responds independently

3. SCORE + SYNTHESIZE — evaluator scores each trace for:
                          · base surprise (cosine similarity)
                          · Shannon entropy (signal density)
                          · logical flow coherence markers
                        synthesizer clusters consensus sentences into
                        a CRAFT-style Reasoning Knowledge Graph

4. LEDGER STORE       — topology committed (new entry or versioned update)
                        semantic embedding indexed for future retrieval

5. SCAFFOLD INJECT    — topology injected into anchor model as structural
                        hypothesis — final response generated from
                        established reasoning, not raw generation
```

The system maintains a **meta-ledger** — a persistent vector store of
accumulated topologies. Each prompt class builds its own topology over
repeated runs. The architecture is designed so that topology quality
increases monotonically: a higher-scoring synthesis always supersedes a
lower-scoring prior version.

## Configuration

All settings are in `config.yaml`. The key values to review before
first run:

**LLM slots** — set the model for each of the three reasoning slots.
All three default to local Ollama models. Any slot can be switched to
a cloud provider by changing `provider` and supplying the API key under
`api_keys`.

```yaml
llm_slots:
  slot_a:
    provider: ollama
    model: "deepseek-r1:8b"
  slot_b:
    provider: ollama
    model: "ministral-3:8b"
  slot_c:
    provider: ollama
    model: "gemma3:12b"
```

**Anchor model** — which slot generates the final scaffolded response.

```yaml
engine:
  anchor_model_slot: "slot_a"
```

**Ledger** — where topologies are stored and the similarity threshold
for retrieval.

```yaml
ledger:
  storage_path: "./ledger/topology_store"
  similarity_threshold: 0.80
```
First run note: On first use the system downloads the embedding model
(~90MB) from HuggingFace to index your topology ledger. This happens
once and is cached locally. If you change embedding_model in config.yaml
to a larger model, expect a longer initial download.

## Expected Outcome

Verifying Your First Run
Step 1 — Ledger Search:
```No prior topology found — first run for this prompt class.```
Expected on a fresh install. After a second run on the same prompt you should see a
topology ID returned here instead.

Step 2 — Trace Collection:
```[slot_x] Logprobs returned: Yes, tokens: NNN```
Collected 3 trace(s).
Every active slot should respond with "Logprobs returned: Yes." If any slot shows "No
trace returned" the model either isn't pulled in Ollama or timed out. Token count will
vary by model and prompt — that's normal.

Step 3 — Scoring and Synthesis:
```Topology synthesized — score: 0.81, nodes: 1, consensus: 1.00```
Score above 0.5 is healthy. Nodes promoted will be low on a first run with 3 slots and
a short prompt — this is expected. With more runs on the same prompt class nodes will accumulate.

Step 4 — Ledger Commit:
```Stored topology [ID]... (version 1, score 0.81)```
Version 1 confirms this is a new entry. On subsequent runs the version number increments
if the new score is higher.

Step 5 — Scaffold Injection:
```Anchored response generated (copy_score: 0.NNN)```
Pipeline complete — NNNms
Copy score between 0.3 and 0.8 is normal — it measures word overlap between the scaffold
and the response. The RESPONSE block should contain coherent text directly addressing the prompt.

## What This Is a Sample Of

This repository isolates the **Python approximation scorer** from a
larger hybrid system. The full research system replaces the evaluator
with a six-gate thermodynamic scoring pipeline derived from information
theory and physical law — measuring information cost, signal entropy,
conceptual variety, throughput efficiency, flow resistance, and
collective coherence across the model ensemble.

Empirical validation of that system across 1,530 pipeline runs at two
temperature settings (0.2, 0.4, and 0.7) across 21 structured prompts produced
the following findings:

- **52% of prompts converged** (reached gate score stability via early
  stopping) at temperature 0.2 versus 38% at temperature 0.4 —
  demonstrating that higher sampling temperature measurably reduces the
  system's ability to reach collective consensus
- **Gate 4 (throughput efficiency)** was the primary convergence limiter
  at 26% of all convergence events — correctly identifying overthinking
  as the dominant failure mode in the prompt corpus
- **Eight prompts never converged** across all runs at both temperatures,
  mapping precisely to prompts grounded in known cognitive bias literature
  (conjunction fallacy, exponential reasoning traps, genuinely ambiguous
  argument structures) — the system correctly identified contested
  reasoning ground as non-convergent
- **Some prompt types converged faster at higher temperature** — a
  non-obvious finding suggesting that for certain problem classes,
  sampling variance accelerates rather than disrupts consensus formation

This implementation produces the topology structures that the full system
uses as gate scoring inputs. It is both a standalone tool and the
interpretable substrate of a more sophisticated pipeline.

## Core Concepts

1. Non-parametric reasoning trace evaluation
The system scores reasoning quality without weight updates, fine-tuning, or retraining. Evaluation
is a runtime process, not a training process. This is a fundamental architectural distinction from
how LLMs are currently improved.
3. Multi-model consensus as a quality signal
No single model is trusted as authoritative. Quality emerges from the agreement structure across an
ensemble. Surprise and consensus are measured relative to the group, not against a ground truth label.
5. Reasoning topology persistence
Reasoning doesn't disappear after inference. It accumulates into a structured graph that improves over
repeated runs. The system has memory of how it has reasoned before — not as raw text, but as scored,
synthesized structure.
7. Scaffold injection as hypothesis, not answer
Prior topology is injected as a structural constraint on future reasoning, explicitly framed as a
hypothesis to be built upon rather than an answer to be reproduced. The distinction matters — it
prevents the system from collapsing into retrieval.
9. Convergence as a measurable property of reasoning
The system has a quantitative definition of when reasoning has stabilized. Convergence is not
declared by a human — it is detected by the gate scoring pipeline across consecutive windows.
Some questions converge. Some don't. Both outcomes are informative.
11. Failure mode detection without supervision
Gate 4 (throughput efficiency) detects overthinking. Gate 3 (variety adequacy) detects underthinking.
These are structural failures in the reasoning process that the system identifies and measures without
being told what the correct answer is.
13. Temperature sensitivity as a system-level property
The relationship between sampling temperature and consensus formation is measurable across the full
pipeline — not just at the token level. Higher temperature demonstrably reduces convergence rate.
This is a system-level finding, not a model-level one.
15. Non-convergence as signal, not failure
Eight prompts never converged across 1,194 runs. The system correctly identifies contested reasoning
ground as structurally different from resolvable reasoning ground. This is not a bug. It is the system
reporting genuine epistemic uncertainty.
17. Symmetric architecture by design
The six-gate scoring is designed to be applied to human reasoning traces with the same pipeline as
machine traces. The bidirectional extension isn't a future add-on — the architecture was designed for
it from the start. The single-sided implementation is a deliberate first step.

## Planned Research

The following directions are actively in development or planned as the
next phase of this research:

**Bidirectional gate scoring** — extending the scoring pipeline to
evaluate not only what the models produce but the structure of the human
input itself. A user's prompt is a reasoning trace. Embedding it and
measuring its structural relationship to the accumulated topology would
allow the system to detect whether the incoming question is approaching
a well-understood problem from an unexpected angle — and adjust the
scaffold accordingly.

**Topology mitosis** — the fractal engine's mitosis signal (triggered
when surprise exceeds a threshold) currently flags a topology for review.
The planned extension is to allow high-surprise topologies to spawn
sub-topologies rather than simply flagging — creating a branching
knowledge structure that preserves novel reasoning paths rather than
collapsing them into the consensus.

**Curiosity-driven exploration** — a seventh gate that operates as a
background process, autonomously identifying regions of the topology space
with high information gain potential and generating exploratory prompts to
fill them. The system identifies what it does not yet know and asks.

**Outcome-observation loops** — grounding topology edges in causal
feedback from real-world actions. Currently the system reasons about
outcomes. The planned extension connects reasoning traces to observed
results, allowing the topology to encode not just what models agreed on
but what was subsequently verified.

**Hierarchical planning** — using accumulated topology chains as the
basis for long-horizon planning. High-confidence topology sequences
become planning templates. The system learns to recognize structural
patterns across problem classes and apply them compositionally.

## Notes

The three-slot minimum is not arbitrary. The evaluator's consensus
scoring, entropy comparison, and flow coherence detection are all
relative measures — they require multiple traces to have anything to
compare against. A single trace produces no meaningful surprise score.
Two traces produce a binary comparison. Three traces produce the minimum
meaningful consensus signal.

The similarity threshold in the ledger (default 0.80) controls how
aggressively prior topologies are retrieved. Lower thresholds retrieve
more broadly but risk injecting scaffolds from insufficiently similar
prior problems. Higher thresholds retrieve only very close matches but
may leave the system without a scaffold even when relevant prior
experience exists. The appropriate value is domain-dependent and should
be tuned against your specific prompt corpus.
